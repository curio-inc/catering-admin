# ケータリング注文管理システム 要件定義書

- 対象リポジトリ: `curio-inc/catering-admin`
- 公開デモ: https://repit-lp.vercel.app/admin/orders
- 版数: 1.0
- 作成日: 2026-06-17

---

## 1. 概要

本システムは、ケータリング（お弁当・オードブル等）事業者向けの **注文管理 Web アプリケーション** である。
店舗側の管理画面（注文・売上・請求書・設定）と、顧客側の注文受付ページで構成される。

現状のリポジトリは **デモ（営業・提案用）専用ビルド** であり、データベースには接続せず、
サンプルデータ（モックデータ）と各ブラウザの `localStorage` で動作する。本書では、
デモで実装済みの機能を「現状仕様」、本番運用時に必要となる事項を「本番化に向けた課題」として整理する。

---

## 2. 目的・背景

| 項目 | 内容 |
| --- | --- |
| 目的 | 受注〜お届け〜請求までの一連の業務を 1 画面で管理できるようにし、紙・表計算による管理を置き換える |
| 当面の用途 | 顧客（事業者）への提案・操作感の確認を目的とした公開デモ |
| 想定業種 | ケータリング、仕出し弁当、オードブル、社食デリバリー等 |

---

## 3. 用語定義

| 用語 | 定義 |
| --- | --- |
| 注文（Order） | 顧客 1 件分の発注。注文番号（例: `DM-202605-001`）で識別 |
| 注文明細（Order Item） | 注文に含まれる商品の 1 行（商品名・単価・数量・備考） |
| 受け取り方法 | 「店頭受け取り（ピックアップ）」または「デリバリー（配送）」 |
| 支払い方法 | クレジットカード / 請求書払い / 現金払い |
| UI ステータス | 画面上の進捗区分（新規 / 対応中 / 完了 / キャンセル） |
| 受付停止 | 特定日時を注文不可にする設定 |
| ブランド名 | 画面・メール・帳票に表示する事業者名（環境変数で変更可） |

## 5. 利用者・ロール

| ロール | 説明 | 利用画面 |
| --- | --- | --- |
| 管理者（店舗スタッフ） | 注文の確認・進捗更新・請求書/領収書発行・受付設定を行う | `/admin/*` |
| 顧客（発注者） | 商品を選択し注文を送信する | `/order` |

> **管理画面の認証は単一パスワードによる「ログイン済み / 未ログイン」のみで確定。** 複数ユーザーアカウント・ロール（権限）管理は要件外（本番でも実装しない）。

## 6. 機能要件

### 6.1 認証（ログイン / ログアウト）

- `/admin/*` 配下は `middleware.ts` で保護され、未ログイン時は `/admin/login` へリダイレクトする。
- パスワードは環境変数 `ADMIN_PASSWORD`（未設定時は既定値）と照合する。一致時に iron-session の Cookie を発行する。
- セッション有効期限は 8 時間。
- ログイン済みで `/admin/login` を開いた場合は `/admin/orders` へリダイレクトする。

### 6.2 注文管理（一覧）— `panel: orders`

- **当月売上カード**: 指定年月の「お届け予定」注文の合計金額を表示。キャンセルは合計から除外する。
  - 年月選択（`<input type="month">`）、「今月」「先月」ボタンで対象月を切替。
- **注文一覧**: 選択中の年月に **お届け予定** の注文を表形式で表示。
  - 列: 受付（日時・注文番号）／顧客名／お届け日時／決済方法／金額／状態／「開く」リンク。
  - 並び替え: 「新着順（受付日時の降順）」「お届け日順（昇順）」を切替。
  - 状態列のセレクトで UI ステータス（新規 / 対応中 / 完了 / キャンセル）を即時変更でき、`localStorage` に保存される。

### 6.3 注文管理（カレンダー）— `panel: orders-calendar`

- 月カレンダー上に、各日の **お届け予定** 注文を表示する。
- 月送り（前月 / 翌月）が可能。
- 日付内の注文をクリックすると注文詳細（`/admin/orders/[id]`）へ遷移する。

### 6.4 注文詳細 — `/admin/orders/[id]`

- 注文 1 件の全情報を表示する。
  - 受け取り方法、支払い方法（請求書払いの場合は会社名・請求先住所）、お客様情報（氏名・電話・メール、デリバリー時は住所）、備考、案件名・案件番号。
  - 注文明細テーブル（商品 / 数量 / 単価 / 備考 / 小計）。
- **編集可能項目**: お届け日（`date`）・受け取り時間（30 分枠セレクト）。保存すると `localStorage` に反映され、一覧・カレンダーにも反映される。
- **UI ステータス変更**: 詳細画面からも変更可能。
- **領収書発行ボタン**: クレジットカード以外（請求書払い・現金払い）の場合に表示し、印刷用ページへ遷移する（クレカは Stripe 自動送付のため非表示）。

### 6.5 請求書発行 — `panel: invoices`

- **対象**: 支払い方法が「請求書払い」の注文のみ一覧表示する。
- 列: 注文No.（受付日時）／顧客名／お届け日時／金額（税込）／操作。
- **操作**:
  - **PDF 出力**: 請求書（`InvoiceFormalDocument`）をクライアント側で PDF 化しダウンロードする。ファイル名はブランド名・注文番号から生成。
  - **自動送信**: 請求書を顧客メール宛に送信した体裁の操作。送信済みは `localStorage` に記録し「送信済み」バッジを表示する（デモでは実際の送信は行わない）。
- URL パラメータ（`?panel=invoices&order=<id>`）指定時は該当注文の請求書プレビューを表示する。

### 6.6 領収書発行（印刷）— `/admin/orders/[id]/receipt-print`

- 正式な領収書レイアウト（`ReceiptFormalDocument`）を A4 想定で表示する。
- 印刷用ツールバー（印刷・戻る）を備える。
- クレジットカード決済の注文では表示せず 404 とする（Stripe 自動送付のため）。

### 6.7 通知メールテンプレ — `panel: email-template`

- 業務で送信される通知メールのプレビューを、タブ切替で確認できる（サンプルデータベース）。
  - **店舗向け**: 新規注文通知（管理者宛）。件名「【新規注文】〇〇 様からのご注文」。
  - **お客様向け**: 受付確認メール。
  - **キャンセル**: 注文者宛キャンセル通知。
  - **領収書**: クレジットカード（Stripe）決済完了時の自動配信領収書メール。
- 各タブで件名と本文プレビューを表示する。

### 6.8 設定（受付停止）— `panel: settings`

- 注文を受け付けない日時（受付停止）を登録する。
  - 行ごとに「受付停止日」「時間（開始）」「時間（終了）」を設定。
  - 行の追加 / 削除、保存（`localStorage` 保存）が可能。
- 設定内容は顧客注文ページの受付可否判定に連動する。

### 6.9 顧客向け注文ページ — `/order`

- **商品選択**: カテゴリ別の商品カードから個数を入力。小計・消費税（8%）・合計を自動計算する。
- **受け取り方法**: 店頭受け取り / デリバリー。デリバリー時は住所・到着時対応（到着後TEL / 置き配 / その他）を入力。
- **お客様情報**: 会社名・氏名・電話・メール、受取日・受取時間（30 分枠）、当日連絡先、備考。
- **支払い方法**: クレジットカード / 請求書払い / 現金払い。請求書払い時は請求先情報（会社名・案件番号・案件名・メール・住所・備考）を入力。
- **受付停止チェック**: 管理画面で設定した受付停止日時に該当する場合、注文を受け付けない旨を表示する。
- 送信後は受付完了の表示を行う（デモでは実データ保存・メール送信は行わない）。

## 7. データモデル（デモ）

### 7.1 注文（`OrderRow`）主要項目

| 項目 | 説明 |
| --- | --- |
| `id` / `order_number` | 注文 ID / 注文番号（例: `DM-202605-001`） |
| `created_at` / `updated_at` | 受付日時 / 更新日時 |
| `status` | 業務ステータス（`requested` / `confirmed` / `preparing` / `delivered` / `cancelled`） |
| `payment_status` | 入金状況（`paid` / `unpaid`） |
| `subtotal_yen` / `tax_yen` / `total_yen` | 小計 / 消費税 / 合計 |
| `payment_method` | 支払い方法（`invoice` / `credit_card` / `cash`） |
| `invoice_company_name` / `invoice_billing_address` | 請求先会社名 / 請求先住所 |
| `customer_name` / `customer_email` / `customer_phone` / `customer_address` | 顧客情報 |
| `delivery_date` / `delivery_time` | お届け日 / お届け時間 |
| `notes` / `management_number` | 備考 / 案件番号 |
| `invoice_issued_at` / `invoice_sent_at` | 請求書 発行日時 / 送信日時 |

### 7.2 注文明細（`OrderItemRow`）

`menu_name_snapshot`（商品名）/ `unit_price_yen_snapshot`（単価）/ `quantity`（数量）/ `custom_text`（備考）/ `sort_order`。

### 7.3 業務ステータス → UI ステータス対応

| 業務ステータス | UI ステータス |
| --- | --- |
| `requested` | 新規 |
| `confirmed` / `preparing` | 対応中 |
| `delivered` | 完了 |
| `cancelled` | キャンセル |

### 7.4 クライアント永続化（`localStorage` キー）

| キー | 用途 |
| --- | --- |
| `catering-admin-demo-order-status` | 注文ごとの UI ステータス上書き |
| `catering-admin-demo-order-delivery` | お届け日時の上書き |
| `catering-admin-demo-invoice-sent` | 請求書送信済みフラグ |
| `catering-admin-reception-stops` | 受付停止設定 |

## 8. 非機能要件

| 区分 | 要件 |
| --- | --- |
| 表示言語 | 日本語（UI・帳票・メールすべて） |
| タイムゾーン | Asia/Tokyo（日付・時刻の集計・表示すべて） |
| 通貨 / 税 | 日本円（¥）/ 消費税 8%（顧客ページの試算） |
| 時間枠 | 受取・お届け時間は 30 分単位 |
| レスポンシブ | PC・タブレットでの操作を想定 |
| アクセシビリティ | 見出し・`aria-*` 属性・`role`・`sr-only` ラベルを付与 |
| ブランド可変 | ブランド名・請求書発行元情報を環境変数で差し替え可能 |

### 主な環境変数

| 変数 | 用途 |
| --- | --- |

| `SESSION_SECRET` | セッション暗号化キー |

## 9. 制約・前提（デモ仕様）

1. データベース・外部 API には接続せず、注文データは `lib/mock-orders.ts` の固定サンプルである。
2. ステータス変更・お届け日時編集・請求書送信状態は **閲覧中ブラウザの `localStorage` のみ** に保存され、他端末・他ユーザーには共有されない。
3. 請求書「自動送信」「注文送信」「メール送信」は **実際の送信を行わない**（UI 上の演出のみ）。
   - クレジットカード決済（Stripe）は **採用可否が未定**。画面・メールテンプレはクレカ前提で用意済みだが、実際の決済処理は行わない。
4. 認証は単一パスワードのみ（**この方式で確定。** ユーザー個別アカウント・ロール管理は要件外）。
5. PDF 生成はクライアント（ブラウザ）側で行う。

## 10. 本番化に向けた課題（参考）

| 項目 | 内容 |
| --- | --- |
| データ永続化 | モックデータ／`localStorage` を実 DB（例: Supabase / PostgreSQL）へ置き換え |
| メール送信 | 注文確認・キャンセル・請求書・領収書メールの実送信（メール配信基盤との連携） |
| 決済連携（**未定**） | クレジットカード決済（Stripe）の採用可否が未確定。導入する場合は決済処理と領収書自動配信を実装する。現状の UI／メールテンプレ／`needsManualReceipt` 判定はクレカ前提で用意済み |
| 監査・履歴 | ステータス変更履歴、操作ログの記録 |
| 顧客ページの受注確定 | 注文送信時のサーバー保存・在庫/受付枠の整合制御 |

> **認証方式は単一パスワードで確定済みのため、ユーザーアカウント・ロール管理は本番化課題に含めない。**
